# Project Architecture Overview

## Product
Athlete Social Network (“The Athlete Passport”): a LinkedIn-style network for athletes focused on verifiable physical and medical records.

## Regulatory posture
- Colombian Habeas Data (Ley 1581 de 2012)
- Future-ready for GDPR-style controls

## Core principle
The architecture is intentionally rigid because most code will be generated by AI agents.

## Applications
- `apps/api` — Fastify + tRPC v11 backend
- `apps/web` — Next.js App Router public web / SEO
- `apps/mobile` — Expo + React Native with expo-router

## Shared packages
- `packages/validators` — Zod schemas and inferred types
- `packages/auth` — Supabase auth wrapper and hooks
- `packages/crypto` — `encryptPII()` and `decryptPII()` only
- `packages/api-client` — typed tRPC client + hooks
- `packages/shared-logic` — pure TS utilities, constants, converters, formatters
- `packages/queue` — BullMQ queue definitions and job payload types

## Monorepo principles
- apps are isolated from each other
- packages are imported by apps, not vice versa
- UI is app-specific
- contracts are schema-first
- OCR is human-verified before display
- RLS is tested and treated as a first-class security boundary

## Operational spine
Zod → tRPC → Prisma

## Data flows
### Public athlete profile
Next.js Server Component → tRPC `createCallerFactory` → explicit Prisma `select` → public-safe output schema

### Private medical document
Mobile upload request → `storageRouter` signed upload flow → storage confirmation → BullMQ OCR job → `PENDING_REVIEW` → reviewer verification → `verifiedData`
